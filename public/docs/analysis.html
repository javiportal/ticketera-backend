Sección 1

Descripción del Sistema

Ticketera es una plataforma web de gestión de eventos y venta de entradas que expone una API RESTful protegida con autenticación basada en tokens (Laravel Sanctum). El sistema implementa un modelo de control de acceso basado en roles (RBAC) con tres perfiles principales:

Rol Permisos Descripción
Admin manage-users, manage-events, validate-tickets, view-reports, sell-tickets Gestión completa del sistema
Organizer manage-events, validate-tickets, view-reports Crear y gestionar eventos propios
Client sell-tickets Navegar eventos, comprar entradas

Relaciones Clave

Relación Tipo Descripción
users → events 1:N Un usuario (organizador) puede crear múltiples eventos
users → tickets 1:N Un usuario (cliente) puede comprar múltiples entradas
events → ticket_types 1:N Cada evento define varios tipos de entrada (General, VIP, etc.)
ticket_types → tickets 1:N Cada tipo de entrada puede tener múltiples tickets vendidos
tickets → attendances 1:1 Cada ticket puede tener un registro de asistencia
users ↔ roles N:M Relación muchos a muchos a través de role_user
roles ↔ permissions N:M Relación muchos a muchos a través de permission_role

Flujograma: Registro y Autenticación

Flujo completo del proceso de registro de nuevos usuarios e inicio de sesión con tokens Sanctum.

flowchart TD
    A([Inicio]) --> B{¿Tiene cuenta?}

    B -- No --> C["POST /api/register"]
    C --> D{¿Datos válidos?}
    D -- No --> E["422 Error de validación"]
    E --> C
    D -- Sí --> F["Crear usuario con rol client"]
    F --> G["Generar token Sanctum"]
    G --> H["201 Retornar token + usuario"]

    B -- Sí --> I["POST /api/login"]
    I --> J{¿Credenciales correctas?}
    J -- No --> K["401 Credenciales inválidas"]
    K --> I
    J -- Sí --> L{¿Cuenta activa?}
    L -- No --> M["403 Cuenta desactivada"]
    L -- Sí --> N["Generar token Sanctum"]
    N --> O["200 Retornar token + usuario"]

    H --> P["Usar token en header Authorization"]
    O --> P
    P --> Q["Acceder a endpoints protegidos"]
    Q --> R{¿Cerrar sesión?}
    R -- No --> Q
    R -- Sí --> S["POST /api/logout"]
    S --> T["Revocar token"]
    T --> U([Fin])

Flujograma: Compra de Entradas

Proceso de navegación de eventos y adquisición de entradas por parte de un cliente autenticado.

flowchart TD
    A(["Cliente autenticado"]) --> B["GET /api/events"]
    B --> C["Ver eventos activos"]
    C --> D["GET /api/events/{id}"]
    D --> E["Ver detalle + tipos de entrada"]
    E --> F{¿Desea comprar?}
    F -- No --> C

    F -- Sí --> G["POST /api/tickets"]
    G --> H{¿Tiene rol client?}
    H -- No --> I["403 Forbidden"]

    H -- Sí --> J{¿Evento activo?}
    J -- No --> K["422 Evento inactivo"]

    J -- Sí --> L{¿Tipo pertenece al evento?}
    L -- No --> M["422 Tipo inválido"]

    L -- Sí --> N{¿Quedan entradas?}
    N -- No --> O["422 Agotadas"]

    N -- Sí --> P{¿Ya compró este tipo?}
    P -- Sí --> Q["422 Duplicado"]

    P -- No --> R["Crear ticket con UUID"]
    R --> S["Estado: valid"]
    S --> T["201 Retornar ticket"]
    T --> U["GET /api/tickets"]
    U --> V["Ver historial de compras"]
    V --> W([Fin])

Flujograma: Validación de Entrada en Puerta

Proceso de verificación y validación de entradas al momento del ingreso al evento.

flowchart TD
    A(["Verificador en puerta"]) --> B["Escanear código UUID"]
    B --> C["POST /api/tickets/{code}/validate"]
    C --> D{¿Tiene permiso validate-tickets?}
    D -- No --> E["403 Forbidden"]

    D -- Sí --> F{¿Ticket existe?}
    F -- No --> G["404 No encontrado"]

    F -- Sí --> H{¿Estado del ticket?}
    H -- "cancelled" --> I["422 Ticket cancelado"]
    H -- "used" --> J["422 Ticket ya utilizado"]

    H -- "valid" --> K["Cambiar estado a used"]
    K --> L["Crear registro Attendance"]
    L --> M["Registrar checked_in_at"]
    M --> N["200 Retornar datos de asistencia"]
    N --> O{¿Más tickets?}
    O -- Sí --> B
    O -- No --> P([Fin])

Flujograma: Gestión de Eventos (Admin)

Operaciones CRUD disponibles para administradores y organizadores sobre los eventos del sistema.

flowchart TD
    A(["Admin / Organizador"]) --> B{Acción}

    B -- Listar --> C["GET /api/admin/events"]
    C --> D{¿Es admin?}
    D -- Sí --> E["Ver TODOS los eventos"]
    D -- No --> F["Ver solo eventos propios"]

    B -- Crear --> G["POST /api/admin/events"]
    G --> H{¿Datos válidos?}
    H -- No --> I["422 Error de validación"]
    H -- Sí --> J["Crear evento"]
    J --> K["Crear 3 tipos: General + VIP + Premium"]
    K --> L["201 Retornar evento"]

    B -- Actualizar --> M["PUT /api/admin/events/{id}"]
    M --> N{¿Es dueño o admin?}
    N -- No --> O["403 Forbidden"]
    N -- Sí --> P["Actualizar evento"]
    P --> Q["200 Retornar evento actualizado"]

    B -- Eliminar --> R["DELETE /api/admin/events/{id}"]
    R --> S{¿Es dueño o admin?}
    S -- No --> T["403 Forbidden"]
    S -- Sí --> U["Eliminar en CASCADE"]
    U --> V["200 Evento eliminado"]

    B -- Asistentes --> W["GET /api/admin/events/{id}/attendees"]
    W --> X["Lista paginada de asistentes"]

    E --> Y([Fin])
    F --> Y
    L --> Y
    Q --> Y
    V --> Y
    X --> Y
